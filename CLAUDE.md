# CLAUDE.md — Claude Code Working Rules for SmartFlow

## Startup procedure (before every task)
1. Read PROJECT_STATUS.md for current state and backlog.
2. If context_output.md exists (generated by `kb-load`), read it too. Do not assume it exists.
3. Inspect the actual current implementation of the files involved — do not assume how the code works from memory or from this file.
4. Verify the requested plan matches the real codebase.
5. If there is a conflict between the request and the code:
   - Explain the conflict clearly.
   - Propose a revised plan.
   - Do NOT start coding until the conflict is resolved.

## Architecture rules
- Reuse existing services/functions before creating new ones.
- Do not create duplicate features or parallel systems.
- Prefer extending the existing architecture over rebuilding.
- The agent worker (agent/worker) is the central "Agent Core": auth, memory read, context building, prompt building, model call. Briefing and Chat both build on it — keep shared logic in shared functions, not copied.
- Learn AI is currently a SEPARATE system by deliberate decision. Do NOT merge it into the agent core unless explicitly asked.

## Development rules
- Analyze first, code second.
- Make small, reviewable changes. Do not do large refactors unless explicitly requested.
- After completing a task, do NOT push unless asked — the user reviews locally first.
- Only update PROJECT_STATUS.md when explicitly asked (it is maintained manually; there is no auto-update workflow).
- Explain risks before any major architecture change.
- After changing worker code, remember it must be redeployed (`npx wrangler deploy`); secrets do not need redeploy, code does.

## Critical rule
Never assume the architecture. Always inspect the current implementation first.
